## 1. Summary

The config is exactly as advertised: light, Vimscript-only, and capability-switchable
via the three idioms (`""` Plug lines, `*.vim_saved` modules, `has()`/`exists()` guards).
It boots on Neovim 0.12.2 (confirmed available in this sandbox). The standout structural
finding is a **completion/LSP vacuum**: `coc.nvim` is declared in `plug#begin` but its
settings module is doubly disabled (renamed `coc.vim_saved` *and* wrapped in `if 0`), and
`ALE` is active but has zero linter config in any active module. So the config currently
ships an idle completion engine and an idle lint engine. Meanwhile Neovim shipped a
built-in plugin manager (`vim.pack`) in 0.12, which reshapes the north-star options.

All ten baseline items were verified against the live files. None were obsolete; several
were refined by research (vim-plug is *not* abandoned; deoplete *is* end-of-development;
the LazyVim default fuzzy finder flipped to fzf-lua).

**Counts:** findings new: 3 · suppressed-by-ledger: 0 (first run; nothing prior) ·

## 5. New findings (not in baseline)

- **N1 — `coc.nvim` is loaded but inert (combine with B1).** Because the plugin loads
  (vimrc:109) while its config is `if 0`+`_saved`, the config pays coc's startup cost and
  Node dependency for zero benefit. The cleanest *light* outcome is to pick a lane:
  either re-enable coc deliberately, or `""`-comment the Plug line until LSP is decided.
  Riskier (touches completion lane) → review PR / decision needed.

- **N2 — `vim.pack` (Neovim 0.12 built-in) changes the north-star math.** New since
  training-era assumptions: Neovim core now ships a built-in plugin manager. For a config
  valued for being *light*, `vim.pack` is arguably a better north star than lazy.nvim —
  zero third-party dependency, maintained by core — at the cost of no event/ft/cmd lazy
  loading. Worth weighing alongside lazy.nvim in the standing recommendation. Riskier.

- **N3 — Stray duplicate files.** `settings/neovide.vim_saved` duplicates the active
  `settings/neovide.vim`; `autoload/plug.vim.old` and `autostart/plug.vim` are stale plug
  copies (see B10). Pure cruft. Low-risk.

## 7. North-star management assessment

**Recommendation (this quarter):** Do **not** rush a full Lua/lazy.nvim rewrite. For a
config explicitly valued for being *light and switchable*, the highest-leverage, lowest-
regret move is to **first resolve the completion/lint vacuum (B1/B2/N1)** — the config
today carries coc + ALE and uses neither. Decide the lane before re-platforming.

On the foundation itself: the owner is Neovim-primary, but the `vimrc` still earns its keep
as a *shared* vim+nvim file (win32 non-nvim branches). A wholesale Vimscript→Lua move buys
native LSP, blink.cmp, treesitter, and the modern ecosystem, but **costs the shared-config
property and is high-effort** — disproportionate for a light setup. If/when the owner
commits to Neovim-only, the cleanest target given Neovim 0.12 is **native LSP + blink.cmp
on `vim.pack`** (core-maintained, zero third-party manager, truest to "light") rather than
lazy.nvim, unless lazy-loading startup time becomes a felt problem. Until that decision,
keep vim-plug and add a `:PlugSnapshot` for reproducibility (B4). This is a standing item;
revisit next quarter.
