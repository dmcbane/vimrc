#### Proposed PR 3: Completion, LSP, and linting — post-coc/ALE path forward
**Branch would be:** `claude/vim-audit-lsp-completion`
**Effort:** high (full migration) or low (just document) · **Risk:** high

With `coc.nvim` and `ALE` both removed, the config currently has:
- No LSP (no go-to-definition, no hover, no rename)
- No completion (beyond Vim's built-in `<C-n>` word completion)
- No linting / formatting (aside from the Elixir `mix format` on save in `settings/elixir.vim`)

**Option A — Native LSP + blink.cmp (Neovim-primary, Lua):**
The 2026 community default for new Neovim configs. Requires adding `settings/lsp.lua`
(or a Vimscript wrapper) and installing language servers (`gopls`, `elixir-ls`, etc.).
`blink.cmp` (v1.10.2, April 2026) is the current LazyVim default completion engine,
with significantly better performance than `nvim-cmp`.

```vimscript
" settings/lsp.plug_saved — rename to .plug to activate
if has('nvim')
  Plug 'neovim/nvim-lspconfig'
  Plug 'saghen/blink.cmp', {'tag': 'v1.*'}
endif
```

**Option B — Add back ALE with linters configured (lighter lift):**
Re-enable ALE from `rails.plug_saved` / `clojure.plug_saved` but add a new
`settings/ale.vim` module that actually configures `g:ale_linters` and
`g:ale_fixers`. No Lua required; works in both Vim and Neovim.

**Option C — Wait (current state):**
Acceptable if Go and Elixir editing are infrequent. vim-go provides its own LSP-adjacent
features (`:GoDef`, `:GoInfo`) via gopls internally, so Go editing may feel complete
even without native LSP.

**Source:** https://github.com/saghen/blink.cmp · https://github.com/neovim/nvim-lspconfig

## 7. North-star management assessment

**Recommendation (this quarter):** The highest-value near-term action is to resolve the
colorscheme emergency (N4 — gruvbox8 archived). That is a 15-minute drop-in replacement.

On the foundation: the config is now considerably cleaner than Q2. The completion/LSP
vacuum is now honest (nothing installed = nothing pretending to work) rather than the
confusing previous state of coc loaded but doubly disabled. The natural next evolution
for a Neovim-primary user is **native LSP + blink.cmp** (Option A above), but this is
high-effort and should be done deliberately, not rushed. The `vim.pack` built-in manager
is **not available in Neovim 0.11.1** (it shipped in 0.12, released March 29, 2026) —
once the owner upgrades to 0.12, it becomes an option worth revisiting.

**Standing recommendation (unchanged):** Keep vim-plug for now; it works. Add a
`:PlugSnapshot` when possible (B4). If/when committing to Neovim-only, the cleanest
target is native LSP + blink.cmp on `vim.pack` (zero third-party manager). The
Vimscript→Lua rewrite (B5) is a companion item — high-effort, high-payoff, but only
worthwhile after the LSP decision is made.
